# 失败熔断

失败熔断（Circuit Breaker）是一种自动保护机制，当任务连续失败达到设定阈值时，自动暂停任务一段时间，避免持续失败的任务占用系统资源，保护系统稳定性。

## 为什么需要失败熔断

### 问题场景

在实际生产环境中，任务可能因为各种原因持续失败：

1. **外部服务不可用**：依赖的 API、数据库等服务宕机
2. **网络故障**：网络连接不稳定或中断
3. **配置错误**：错误的配置导致任务无法执行
4. **资源耗尽**：内存、磁盘等资源不足
5. **代码缺陷**：程序 bug 导致必然失败

**没有熔断机制的后果：**
- 持续失败的任务浪费系统资源
- 重试机制导致资源消耗倍增
- 影响其他正常任务的执行
- 大量错误日志占用磁盘空间
- 告警系统被淹没

### 熔断机制的价值

- **自动保护**：自动识别并隔离问题任务
- **资源优化**：释放资源给正常任务使用
- **快速恢复**：问题解决后自动恢复运行
- **降低噪音**：减少错误日志和告警数量
- **运维友好**：给予运维人员处理问题的时间

## 熔断机制原理

### 工作流程

```
正常状态 → 失败累积 → 达到阈值 → 触发熔断 → 自动暂停 → 等待恢复 → 自动恢复
```

**详细说明：**

1. **正常执行**：任务按计划正常执行
2. **失败记录**：任务失败后，记录失败次数
3. **阈值检查**：检查失败次数是否达到阈值
4. **触发熔断**：达到阈值后，自动暂停任务
5. **暂停期**：任务在指定时间内不会执行
6. **自动恢复**：暂停期结束后，自动恢复任务执行
7. **重置计数**：任务成功执行后，重置失败计数

### 状态图

```
         [正常执行]
              |
         失败次数 < 阈值
              |
         [继续执行]
              |
         失败次数 >= 阈值
              |
         [触发熔断]
              |
         [自动暂停]
              |
        等待暂停时间
              |
         [自动恢复]
              |
         执行成功
              |
         [重置计数]
```

## 配置参数

熔断机制通过 `JobOptions` 中的三个参数配置：

### FailThreshold - 失败阈值

**类型：** `int`

**说明：** 连续失败多少次后触发熔断。设置为 0 表示不启用熔断。

**推荐值：**
- 网络请求任务：3-5 次
- 数据处理任务：5-10 次
- 轻量级任务：2-3 次

**示例：**

```go
cron.JobOptions{
    FailThreshold: 5, // 连续失败5次触发熔断
}
```

### FailWindow - 失败时间窗口

**类型：** `time.Duration`

**说明：** 统计失败的时间窗口。设置为 0 表示不限时间窗口（累计统计）。

**推荐值：**
- 高频任务（每分钟）：5-10 分钟
- 中频任务（每小时）：1-2 小时
- 低频任务（每天）：不设置（累计）

**示例：**

```go
cron.JobOptions{
    FailThreshold: 5,
    FailWindow:    10 * time.Minute, // 10分钟内失败5次触发熔断
}
```

**时间窗口的作用：**
- 避免偶发失败触发熔断
- 关注近期失败情况
- 自动过期老旧失败记录

### PauseDuration - 暂停时长

**类型：** `time.Duration`

**说明：** 触发熔断后暂停多长时间。默认值：30 秒。

**推荐值：**
- 短暂故障：1-5 分钟
- 外部服务故障：10-30 分钟
- 严重问题：1-2 小时

**示例：**

```go
cron.JobOptions{
    FailThreshold: 5,
    FailWindow:    10 * time.Minute,
    PauseDuration: 30 * time.Minute, // 暂停30分钟
}
```

## 使用示例

### 示例1：基础配置

```go
package main

import (
    "context"
    "errors"
    "fmt"
    "math/rand"
    "time"

    "github.com/darkit/cron"
)

func main() {
    c := cron.New()

    // 模拟不稳定的任务
    c.Schedule("unstable-task", "*/5 * * * * *", func(ctx context.Context) {
        // 70% 概率失败
        if rand.Float64() < 0.7 {
            panic("任务执行失败")
        }
        fmt.Println("任务执行成功")
    }, cron.JobOptions{
        FailThreshold: 3,                 // 连续失败3次触发熔断
        FailWindow:    1 * time.Minute,   // 1分钟时间窗口
        PauseDuration: 30 * time.Second,  // 暂停30秒
        MaxRetries:    0,                 // 不重试
    })

    c.Start()
    defer c.Stop()

    // 运行3分钟观察熔断效果
    time.Sleep(3 * time.Minute)
}
```

### 示例2：API 调用任务

```go
package main

import (
    "context"
    "fmt"
    "net/http"
    "time"

    "github.com/darkit/cron"
)

type APICheckJob struct {
    url string
}

func (j *APICheckJob) Name() string {
    return "api-health-check"
}

func (j *APICheckJob) Run(ctx context.Context) error {
    req, err := http.NewRequestWithContext(ctx, "GET", j.url, nil)
    if err != nil {
        return err
    }

    resp, err := http.DefaultClient.Do(req)
    if err != nil {
        return err
    }
    defer resp.Body.Close()

    if resp.StatusCode != http.StatusOK {
        return fmt.Errorf("API返回状态码: %d", resp.StatusCode)
    }

    fmt.Printf("API健康检查通过: %s\n", j.url)
    return nil
}

func main() {
    c := cron.New()

    job := &APICheckJob{
        url: "https://api.example.com/health",
    }

    // 配置熔断保护
    c.ScheduleJobByName("@every 30s", job, cron.JobOptions{
        Timeout:       10 * time.Second,  // 超时10秒
        FailThreshold: 5,                 // 连续失败5次触发熔断
        FailWindow:    5 * time.Minute,   // 5分钟时间窗口
        PauseDuration: 15 * time.Minute,  // 暂停15分钟
        MaxRetries:    2,                 // 失败后重试2次
        RetryInterval: 5 * time.Second,   // 重试间隔5秒
    })

    c.Start()
    defer c.StopGracefully(10 * time.Second)

    // 持续运行
    select {}
}
```

### 示例3：数据库同步任务

```go
package main

import (
    "context"
    "fmt"
    "time"

    "github.com/darkit/cron"
)

type DBSyncJob struct {
    sourceDB string
    targetDB string
}

func (j *DBSyncJob) Name() string {
    return "db-sync"
}

func (j *DBSyncJob) Run(ctx context.Context) error {
    fmt.Printf("开始同步数据: %s -> %s\n", j.sourceDB, j.targetDB)

    // 模拟数据库同步
    // 实际应用中这里是真实的数据库操作
    select {
    case <-time.After(2 * time.Second):
        fmt.Println("数据同步完成")
        return nil
    case <-ctx.Done():
        return ctx.Err()
    }
}

func main() {
    c := cron.New()

    syncJob := &DBSyncJob{
        sourceDB: "main-db",
        targetDB: "backup-db",
    }

    // 数据库同步任务的熔断配置
    c.ScheduleJobByName("@every 10m", syncJob, cron.JobOptions{
        Timeout:       5 * time.Minute,   // 超时5分钟
        FailThreshold: 3,                 // 连续失败3次触发熔断
        PauseDuration: 1 * time.Hour,     // 暂停1小时给运维处理
        MaxRetries:    2,                 // 重试2次
        RetryInterval: 1 * time.Minute,   // 重试间隔1分钟
        Async:         true,              // 异步执行
    })

    c.Start()
    defer c.StopGracefully(10 * time.Second)

    // 持续运行
    select {}
}
```

## 参数配置建议

### 不同场景的配置策略

#### 1. 网络请求任务

```go
// 特点：失败快，恢复快
cron.JobOptions{
    FailThreshold: 5,                 // 允许5次失败
    FailWindow:    5 * time.Minute,   // 5分钟时间窗口
    PauseDuration: 10 * time.Minute,  // 暂停10分钟
    MaxRetries:    3,                 // 每次失败重试3次
    RetryInterval: 10 * time.Second,  // 重试间隔10秒
}
```

#### 2. 数据处理任务

```go
// 特点：失败影响大，需要人工介入
cron.JobOptions{
    FailThreshold: 3,                 // 仅允许3次失败
    FailWindow:    1 * time.Hour,     // 1小时时间窗口
    PauseDuration: 2 * time.Hour,     // 暂停2小时
    MaxRetries:    1,                 // 仅重试1次
    RetryInterval: 5 * time.Minute,   // 重试间隔5分钟
}
```

#### 3. 监控任务

```go
// 特点：频率高，允许偶发失败
cron.JobOptions{
    FailThreshold: 10,                // 允许10次失败
    FailWindow:    10 * time.Minute,  // 10分钟时间窗口
    PauseDuration: 5 * time.Minute,   // 暂停5分钟
    MaxRetries:    0,                 // 不重试
}
```

#### 4. 批处理任务

```go
// 特点：执行时间长，失败代价高
cron.JobOptions{
    FailThreshold: 2,                 // 仅允许2次失败
    PauseDuration: 4 * time.Hour,     // 暂停4小时
    MaxRetries:    1,                 // 仅重试1次
    RetryInterval: 30 * time.Minute,  // 重试间隔30分钟
    Timeout:       1 * time.Hour,     // 超时1小时
}
```

## 熔断状态监控

### 使用事件钩子监控

```go
package main

import (
    "fmt"
    "log"
    "time"

    "github.com/darkit/cron"
)

func main() {
    // 创建事件钩子记录熔断
    hook := func(ev cron.Event) {
        if ev.End.IsZero() {
            return // 忽略开始事件
        }

        if !ev.Success {
            log.Printf("[失败] 任务 %s 执行失败: %s (重试%d次)",
                ev.TaskID, ev.Error, ev.Retries)
        }
    }

    c := cron.New(cron.WithEventHook(hook))

    // 添加任务
    c.Schedule("monitored-task", "@every 30s", func(ctx context.Context) {
        // 任务逻辑
    }, cron.JobOptions{
        FailThreshold: 5,
        PauseDuration: 10 * time.Minute,
    })

    c.Start()
    defer c.Stop()

    select {}
}
```

### 自定义监控

```go
package main

import (
    "context"
    "fmt"
    "time"

    "github.com/darkit/cron"
)

type CircuitBreakerMonitor struct {
    scheduler *cron.Cron
}

func (m *CircuitBreakerMonitor) Check() {
    ticker := time.NewTicker(30 * time.Second)
    defer ticker.Stop()

    for range ticker.C {
        stats := m.scheduler.GetAllStats()

        for taskID, stat := range stats {
            // 检查是否被暂停
            if stat.IsPaused {
                pauseUntil := stat.PauseUntil
                if !pauseUntil.IsZero() && time.Now().Before(pauseUntil) {
                    remaining := time.Until(pauseUntil)
                    fmt.Printf("[熔断] 任务 %s 已暂停，还需等待 %v\n",
                        taskID, remaining.Round(time.Second))
                }
            }

            // 检查失败率
            if stat.RunCount > 0 {
                failRate := float64(stat.FailCount) / float64(stat.RunCount)
                if failRate > 0.5 {
                    fmt.Printf("[警告] 任务 %s 失败率过高: %.2f%%\n",
                        taskID, failRate*100)
                }
            }
        }
    }
}

func main() {
    c := cron.New()

    // 添加任务...

    c.Start()
    defer c.Stop()

    // 启动监控
    monitor := &CircuitBreakerMonitor{scheduler: c}
    go monitor.Check()

    select {}
}
```

## 手动恢复和自动恢复

### 自动恢复

熔断触发后，任务会在 `PauseDuration` 时间后自动恢复：

```go
c.Schedule("task", "@every 1m", handler, cron.JobOptions{
    FailThreshold: 5,
    PauseDuration: 10 * time.Minute, // 10分钟后自动恢复
})
```

**自动恢复流程：**
1. 失败次数达到阈值，任务被暂停
2. 等待 `PauseDuration` 时间
3. 时间到达后，自动恢复任务执行
4. 失败计数器重置为 0
5. 重新开始统计失败次数

### 手动恢复

使用 `Resume()` 方法可以提前手动恢复任务：

```go
// 手动恢复被熔断的任务
if err := c.Resume("task-id"); err != nil {
    log.Printf("恢复任务失败: %v", err)
} else {
    log.Println("任务已手动恢复")
}
```

**使用场景：**
- 问题已解决，提前恢复
- 运维介入，手动干预
- 测试熔断恢复流程

**[注意]：** 手动恢复会重置失败计数，任务会立即开始执行。

### 恢复后重新熔断

如果恢复后任务继续失败，会再次触发熔断：

```go
// 任务恢复后继续失败
时间轴：
10:00 - 连续失败5次，触发熔断
10:00 - 暂停10分钟
10:10 - 自动恢复
10:11 - 执行失败（计数1）
10:12 - 执行失败（计数2）
10:13 - 执行失败（计数3）
10:14 - 执行失败（计数4）
10:15 - 执行失败（计数5）
10:15 - 再次触发熔断，暂停10分钟
```

## 与其他功能配合

### 与重试机制配合

```go
// 先重试，重试失败后计入熔断统计
c.Schedule("task", "@every 5m", handler, cron.JobOptions{
    MaxRetries:    3,                 // 每次失败重试3次
    RetryInterval: 30 * time.Second,  // 重试间隔30秒
    FailThreshold: 5,                 // 重试后仍失败，累计5次触发熔断
    PauseDuration: 30 * time.Minute,  // 熔断暂停30分钟
})
```

**工作流程：**
1. 任务执行失败
2. 自动重试 3 次
3. 重试全部失败，计入熔断统计（1次）
4. 达到 5 次后触发熔断

### 与 Misfire 策略配合

```go
// 熔断期间错过的执行不会补跑
c.Schedule("task", "@every 10m", handler, cron.JobOptions{
    MisfirePolicy: cron.MisfireSkip,  // 熔断期间错过的执行跳过
    FailThreshold: 3,
    PauseDuration: 1 * time.Hour,
})
```

### 与事件钩子配合

```go
// 监控熔断事件并发送告警
hook := func(ev cron.Event) {
    if !ev.Success && ev.End.After(ev.Start) {
        // 发送告警
        sendAlert(ev.TaskID, ev.Error)
    }
}

c := cron.New(cron.WithEventHook(hook))
```

## 完整示例

### 生产环境配置示例

```go
package main

import (
    "context"
    "fmt"
    "log"
    "time"

    "github.com/darkit/cron"
)

func main() {
    // 创建日志钩子
    logger := cron.NewDefaultLogger()
    eventHook := cron.NewEventLoggerHook(logger)

    c := cron.New(
        cron.WithLogger(logger),
        cron.WithEventHook(eventHook),
    )

    // 外部API调用任务
    c.Schedule("api-sync", "@every 5m", apiSyncFunc, cron.JobOptions{
        Timeout:       30 * time.Second,
        MaxRetries:    3,
        RetryInterval: 10 * time.Second,
        FailThreshold: 5,
        FailWindow:    30 * time.Minute,
        PauseDuration: 1 * time.Hour,
        Async:         true,
    })

    // 数据库备份任务
    c.Schedule("db-backup", "0 0 2 * * *", dbBackupFunc, cron.JobOptions{
        Timeout:       30 * time.Minute,
        MaxRetries:    1,
        RetryInterval: 10 * time.Minute,
        FailThreshold: 2,
        PauseDuration: 4 * time.Hour,
    })

    // 日志清理任务
    c.Schedule("log-cleanup", "0 0 3 * * *", logCleanupFunc, cron.JobOptions{
        Timeout:       10 * time.Minute,
        FailThreshold: 3,
        PauseDuration: 12 * time.Hour,
    })

    c.Start()
    defer c.StopGracefully(1 * time.Minute)

    // 持续运行
    select {}
}

func apiSyncFunc(ctx context.Context) {
    // API同步逻辑
    fmt.Println("执行API同步")
}

func dbBackupFunc(ctx context.Context) {
    // 数据库备份逻辑
    fmt.Println("执行数据库备份")
}

func logCleanupFunc(ctx context.Context) {
    // 日志清理逻辑
    fmt.Println("执行日志清理")
```

## 常见问题

### Q: 熔断期间可以手动触发任务吗？

A: 可以。使用 `RunNow(id)` 可以在熔断期间手动触发任务，但不会影响熔断状态。

### Q: 熔断后失败计数会清零吗？

A: 会。触发熔断后，失败计数器会重置为 0，恢复后重新开始计数。

### Q: 如何禁用熔断功能？

A: 将 `FailThreshold` 设置为 0 或不设置即可：

```go
cron.JobOptions{
    FailThreshold: 0, // 禁用熔断
}
```

### Q: 熔断期间任务会被删除吗？

A: 不会。任务只是被暂停，不会被删除。可以通过 `List()` 查看所有任务。

### Q: FailWindow 和 PauseDuration 有什么区别？

A:
- `FailWindow`：统计失败次数的时间窗口
- `PauseDuration`：触发熔断后的暂停时长

### Q: 任务成功后失败计数会清零吗？

A: 会。任务执行成功后，失败计数器会重置为 0。

### Q: 熔断状态会持久化吗？

A: 不会。调度器重启后，熔断状态会丢失，所有任务恢复正常。

## 最佳实践

1. **合理设置阈值**：根据任务特性设置合适的 `FailThreshold`
2. **使用时间窗口**：对于高频任务，使用 `FailWindow` 避免误触发
3. **给予恢复时间**：`PauseDuration` 应足够长，让问题有时间解决
4. **配合重试**：先重试，重试失败后再计入熔断
5. **监控告警**：使用事件钩子或 Dashboard 监控熔断状态
6. **文档记录**：记录熔断配置的原因和调整历史
7. **测试验证**：在测试环境模拟失败场景，验证熔断效果
